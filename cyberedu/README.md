# CyberEdu — интерактивная платформа курса по информационной безопасности

## Краткое описание

**CyberEdu** — образовательная платформа, в которой пользователь проходит курс по информационной безопасности через структурированные **лекции**, **тесты**, **практические задания**, **AI-наставника** и в итоге получает **сертификат** (PDF). Интерфейс ориентирован на последовательное обучение: модули открываются по мере прогресса, баллы и достижения мотивируют довести курс до конца.

### Фирменный стиль и логотип

В каталоге **`frontend/public/brand/`** находятся векторные активы:

- **`logo-mark.svg`** — знак (щит, буква C, траектория обучения, книга, искра AI), для шапки, favicon-превью и PDF;
- **`logo-full.svg`** — горизонтальная версия с подписью «Информационная безопасность»;
- **`favicon.svg`** — упрощённый знак для вкладки браузера.

React-компоненты: **`frontend/components/brand/brand-logo.tsx`** (`BrandLogoMark`, `BrandLogoFullImg`, `BrandLogoHeaderLink`).

## Главная особенность

**AI не ограничивается ответами в чате** — он **адаптирует лекции** под интересы пользователя из профиля. Если студент увлекается играми, спортом, дизайном или другой сферой, сложные темы ИБ объясняются через **близкие примеры и аналогии**, сохраняя учебную точность. Дополнительно доступен **AI-чат наставник** в контексте лекции и практики (режим подсказок без выдачи готовых ответов на задания).

## Возможности

| Область | Возможности |
|--------|-------------|
| **Аккаунт** | Регистрация и вход, профиль пользователя, поле **интересов** для AI |
| **Курс** | Модули по порядку, **поэтапное открытие** следующего модуля после завершения предыдущего |
| **Лекции** | Текст **блоками** (структурированная разметка), **видео** при наличии в контенте |
| **Тесты** | Проверка **среднего уровня**, автоматическая оценка на **сервере** |
| **Практика** | Разные типы заданий: текст, файл, комбинированные формы, сценарии |
| **Лаборатории** | Учебная **консоль**, разбор **фишингового** письма, **анализ URL**, **криптография**, **анализ логов** и др. |
| **AI** | **Адаптация лекций** (упрощение, примеры, конспект), **чат-наставник** с политикой безопасности |
| **Прогресс** | Баллы по модулю и курсу, визуализация на **профиле**, **достижения** (бейджи) |
| **Сертификат** | Выдача **PDF** после завершения всех модулей, код проверки |
| **Администрирование** | **Админ-панель**: пользователи, модули, контент, проверка практики, отзывы, сертификаты |
| **Развёртывание** | **Только Docker** / **docker-compose**: PostgreSQL, pgAdmin, backend, frontend; опционально **Redis** |

## Стек технологий

### Frontend

- **Next.js** (App Router, Server Components / Actions)
- **TypeScript**
- **Tailwind CSS**
- **Prisma** — ORM и миграции схемы БД, тесно связанные с учебной логикой курса
- **NextAuth** — сессии (JWT), роли `USER` / `ADMIN`

### Backend

- **FastAPI**
- **PostgreSQL**
- **SQLAlchemy** 2.x
- **Alembic** — миграции
- **JWT** — конфигурация для API (`JWT_SECRET_KEY` в compose; развитие REST-интеграций)
- **Pydantic** — схемы запросов/ответов

> Текущая бизнес-логика курса (прогресс, практика, тесты, сертификаты, админка) в основном реализована во **frontend** (Next.js + Prisma). Backend выступает как **сервисный слой** и точка расширения; предусмотрен перенос части логики на FastAPI по мере развития проекта.

### AI

- Совместимый с **OpenAI** API (`OPENAI_API_KEY`, опционально свой `OPENAI_API_BASE_URL` — в т.ч. **DeepSeek** и аналоги)

### DevOps

- **Docker** и **docker-compose**
- **pgAdmin** для администрирования БД
- **Redis** (опционально, профиль `cache` в compose) — для rate limit и кэша ответов AI в будущем

## Пользовательский сценарий

1. **Главная** — знакомство с платформой и программой курса.  
2. **Регистрация** — создание учётной записи.  
3. **Профиль** — ФИО, учебное заведение, **интересы** (важно для AI).  
4. **Курс** — список модулей, доступ к текущему шагу.  
5. **Лекция** — чтение блоков, при желании **AI-адаптация** под интересы.  
6. **Тест** — закрепление материала (проверка на сервере).  
7. **Практика** — задания и лаборатории, при необходимости проверка преподавателем.  
8. **Следующий модуль** — открывается после выполнения требований текущего.  
9. **Сертификат** — после завершения всех модулей — генерация **PDF** и запись в реестре.

## Роли

| Роль | Описание |
|------|----------|
| **Гость** | Просмотр маркетинговой главной, отзывов, страниц входа/регистрации без доступа к курсу |
| **Студент** (`USER`) | Личный кабинет, курс, тесты, практика, AI, сертификат, отзыв после прогресса |
| **Администратор** (`ADMIN`) | Вся функциональность студента **плюс** админ-панель: пользователи, контент, проверка работ, модерация отзывов, сертификаты |

## Демо-доступ

Учётные записи создаются **seed**-скриптом (см. раздел «Запуск»). Типичные данные для демо:

| Роль | Email | Пароль |
|------|-------|--------|
| **Администратор** | `admin@cyberedu.local` | `Admin12345!` |
| **Студент** | `student@cyberedu.local` | `Student12345!` |

В логах `docker compose logs frontend` сообщение **`[auth][error] CredentialsSignin`** чаще всего от **одной неудачной** попытки входа (неверный пароль); запись **`auth.sign_in_success`** относится к **успешной** попытке до или после неё — это не обязательно противоречие.

Используйте только в **изолированной** демо-среде; в production задайте собственные секреты и пароли.

## Запуск (только Docker)

Официальный способ поднять платформу — **docker compose** из каталога **`cyberedu/`**. Запуск Next.js или Postgres **на хосте без контейнеров** для полного стека не поддерживается в этой документации.

```bash
cp .env.example .env
docker compose up --build
```

После сборки контейнер **frontend** при старте выполняет `prisma migrate deploy` и `prisma db seed` (см. `frontend/docker-entrypoint.sh`), затем запускает приложение. Отдельный `docker compose exec … migrate` обычно не нужен.

Если миграции или seed завершились с ошибкой, смотрите логи: `docker compose logs frontend`.

**Почему «сайт не изменился» после правок в коде:** контейнер отдаёт **старый собранный** Next.js, пока вы не пересоберёте образ. Обязательно:

```bash
docker compose build frontend
docker compose up -d --force-recreate frontend
```

Или из каталога `cyberedu/`:

```bash
./scripts/rebuild-frontend.sh
# при «залипании» кэша слоёв:
./scripts/rebuild-frontend.sh --no-cache
```

Первая сборка образа `frontend` (шаг `npm ci` в Dockerfile) может занять **15–25 минут** из‑за загрузки зависимостей — это нормально. Повторные сборки обычно быстрее за счёт кэша BuildKit.

### Режим правок дизайна (без пересборки образа)

Контейнер **`frontend`** отдаёт **уже собранный** бандл Next.js: изменения в `globals.css`, компонентах и страницах **не видны**, пока вы не выполните `docker compose build frontend` (долго при плохой сети к npm).

Для **мгновенного hot reload** при вёрстке и стилях:

1. Оставьте запущенными **postgres** (и при необходимости **backend**): `docker compose up -d postgres backend` (или полный compose без frontend).
2. В каталоге **`cyberedu/`** выполните:

```bash
./scripts/design-live.sh
```

Скрипт **остановит только** сервис `frontend` в Docker, освободит порт **3100** и запустит **`npm run dev`** в `frontend/`. Сохраняйте файлы — страница обновится сама.

Остановка dev-сервера: **Ctrl+C**. Вернуть сайт из Docker:

```bash
docker compose up -d frontend
```

**При сборке Docker-образа `frontend`:** сообщение **`EIDLETIMEOUT`** / **`Idle timeout reached for host 'registry.npmjs.org'`** — нестабильная сеть до npm. В образе заданы увеличенные таймауты (см. `frontend/.npmrc` и `Dockerfile`); повторите `docker compose build frontend`. При частых обрываах попробуйте другую сеть или VPN, либо зеркало registry (для вашего региона).

Если **`DeadlineExceeded`** / **`load metadata for docker.io/library/node:…`** — нет стабильного доступа к **Docker Hub**. Пока сеть лучше, выполните `docker pull node:20-bookworm-slim`, затем снова `docker compose build frontend`; либо задайте **`NODE_IMAGE`** в `.env` на тот же тег из доступного вам registry (см. комментарии в `cyberedu/.env.example`).

Открывайте сайт по адресу **http://localhost:3100** — это контейнер `frontend` из `docker-compose.yml` (проброс **3100**→3000 внутри контейнера), **или** локальный `next dev` после **`./scripts/design-live.sh`** (см. выше). Не запускайте одновременно **контейнер frontend** и **`npm run dev`** на одном и том же порту **3100**. Если заходите с другого устройства по IP, задайте в `cyberedu/.env` **`AUTH_URL`** и **`NEXT_PUBLIC_APP_URL`** на тот же URL, что в адресной строке (см. `cyberedu/.env.example`).

После **успешной** пересборки образа под логотипом и в подвале отображается **время в формате ISO** (например `2026-05-14T12:34:56.789Z`). Если везде написано **`local`**, вы смотрите **не тот** процесс или образ **не пересобирался**.

Миграции **Alembic** у backend (если понадобятся отдельно от образа):

```bash
docker compose exec backend alembic upgrade head
```

### Redis (опционально)

```bash
docker compose --profile cache up --build
```

Профиль **`cache`** поднимает Redis на порту **6379**; в `backend` задайте `REDIS_URL`, если код начнёт его использовать.

## Безопасность (кратко)

- **Пароли** — хеширование (**bcrypt**), открытый пароль не хранится и не возвращается из API.  
- **Роли** — `USER` / `ADMIN` в сессии; админ-маршруты защищены middleware и серверными проверками.  
- **Маршруты** — `/dashboard/*` только для авторизованных; `/admin/*` — только для `ADMIN`.  
- **Файлы практики** — лимит размера, белый список расширений, проверка **магических байтов**, запрет исполняемых и скриптовых расширений в имени, выдача файлов только владельцу (или админу).  
- **AI** — в промпт не попадают эталоны практики и корректные ответы тестов; политика наставника; rate limit на AI- и auth-эндпоинты.  
- **Тесты** — флаг **`isCorrect`** не отдаётся на клиент; оценка только на **backend**; варианты ответов для выбора перемешиваются при отдаче страницы.

Подробный аудит и рекомендации можно оформить отдельным документом в репозитории при необходимости.

## Что можно улучшить дальше

- Расширить **набор лабораторий** и интерактивных сценариев.
